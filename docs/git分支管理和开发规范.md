# Git 分支管理和开发规范 
## 分支管理 
### 长期分支 
- `main/master` - 主分支，只接受 `feature` 分支合并，存储稳定的文档版本
- `test` - 测试分支（可选），用于文档格式检查、链接验证、内容审核等 
### 短期分支 
- `feature/版本号`：生产发布分支 
- `dev/功能描述-姓名英文缩写`：个人功能开发分支 
- `hotfix/问题描述-姓名英文缩写` - 紧急修复分支 
### 管理流程 

#### 学习笔记/文档库流程（推荐）
> 适用于个人学习笔记、知识库等文档管理场景，流程简化，便于持续积累。

1. **日常学习笔记编写**：
   - 从 `feature/版本号` 或 `main` 分支创建 `dev/学科-主题-ShihongGan` 分支（如 `dev/cpp-basic-syntax-ShihongGan`）
   - 在对应学科文件夹下编写学习笔记和文档
   - 完成一个主题或章节后，提交并推送到远程

2. **文档整理与合并**：
   - 定期将 `dev` 分支合并到 `feature/版本号` 分支
   - 合并前检查文档格式、命名规范是否符合要求
   - 合并后更新 `README.md` 更新日志

3. **版本发布**：
   - 当某个学科或主题的学习告一段落时，将 `feature/版本号` 合并到 `main` 分支
   - 在 `README.md` 中记录版本更新内容

4. **文档修正**：
   - 发现文档错误或需要补充时，从 `feature/版本号` 创建 `hotfix/问题描述-ShihongGan` 分支
   - 修复后直接合并回 `feature/版本号` 和 `main`

**注意事项**：
- 学习笔记以持续积累为主，不需要严格的测试流程
- `test` 分支主要用于文档格式检查、链接验证等（可选）
- 建议按学科分类创建 `dev` 分支，便于管理和回顾

#### 软件开发流程（参考）
> 适用于需要严格测试和版本控制的软件开发场景。

1. 从当前的 `feature` 分支新建特定功能的 `dev` 分支； 
2. 开发完成后，统一将 `dev` 分支合并到 `test` 分支测试，测试通过后，合并成新 `feature` 分支； 
3. 对于发生在 `feature` 中的紧急问题，从当前 `feature` 分支新建 `hotfix` 分支进行修复，测试后合并回 `feature` 发布。 
### 文档分类管理
> 按照学科进行分类管理，便于知识体系的构建和检索。

- **学科分类**：在项目根目录下按学科创建文件夹（如 `cpp`、`data-structures`、`linux`）
- **主题细分**：在学科文件夹下可按主题创建子文件夹（如 `cpp/basic-syntax`、`cpp/stl`）
- **文档组织**：
  - 每个学科建议有 `README.md` 作为索引和目录
  - 相关主题的笔记放在对应子文件夹中
  - 使用链接和标签建立文档间的关联

## 提交信息规范 
> 提交信息前缀，可快速定位。 
- `feat`: 新功能 
- `fix`: 修复问题 
- `perf`: 代码优化 
- `docs`: 文档更新 
- `style`: 代码格式调整（不影响代码） 
- `refactor`: 重构 
- `test`: 测试相关 
- `chore`: 构建过程或辅助工具变动

## 文件和文件夹命名规范
> 遵循大厂标准，确保跨平台兼容性和可维护性。

### 文件编码规范
- **统一使用 UTF-8 编码**：所有文档文件必须使用 UTF-8 编码，确保跨平台兼容性和避免乱码问题
- **Git 配置**：已配置 `core.quotepath=false`、`i18n.commitencoding=utf-8`、`i18n.logoutputencoding=utf-8`
- **.gitattributes**：项目根目录已配置 `.gitattributes` 文件，自动处理文件编码和换行符
- **编辑器设置**：建议在编辑器中设置默认编码为 UTF-8，保存文件时确保使用 UTF-8 编码

### 文件夹命名规范
- **统一使用英文命名**：所有文件夹名称必须使用英文，禁止使用中文字符，确保跨平台兼容性
- **使用 kebab-case（小写+连字符）**：多单词使用连字符分隔，如 `data-structures`、`web-development`
- **简洁明确**：文件夹名称应清晰反映其内容，便于快速定位
- **避免特殊字符**：禁止使用系统保留字符 `\/:*?"<>|`，避免空格
- **示例**：
  - ✅ `cpp`、`data-structures`、`linux`、`web-development`
  - ❌ `数据结构`、`C++学习`、`Web Development`（含空格）

### 文件命名规范
- **统一格式**：文件名应遵循统一格式，包含关键信息
- **使用下划线或连字符分隔**：建议使用下划线 `_` 或连字符 `-` 分隔不同字段，避免使用空格
- **时间前置**：重要文档建议将日期放在前面，便于排序和追溯，格式：`YYYY-MM-DD`
- **模块化结构**：文件名由多个字段组成，格式：`日期_项目名_内容描述_版本_作者`
- **禁止特殊字符**：避免使用系统保留字符 `\/:*?"<>|`，确保跨平台兼容性
- **大小写一致性**：建议使用小写字母，保持命名风格一致
- **示例**：
  - ✅ `2025-12-31_project-report_v1.0_shihonggan.md`
  - ✅ `cpp_basic-syntax_notes.md`
  - ✅ `data-structures_binary-tree_summary.md`
  - ❌ `项目报告 2025.md`（含中文和空格）
  - ❌ `C++学习笔记.docx`（含特殊字符）
