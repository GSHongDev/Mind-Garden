# Git 分支管理和开发规范 
## 分支管理 
### 长期分支 
- `main/master` - 主分支，只接受 `feature` 分支合并 
- `test` - 测试分支，用于集成测试 
### 短期分支 
- `feature/版本号`：生产发布分支 
- `dev/功能描述-姓名英文缩写`：个人功能开发分支 
- `hotfix/问题描述-姓名英文缩写` - 紧急修复分支 
### 管理流程 
1. 从当前的 `feature` 分支新建特定功能的 `dev` 分支； 
2. 开发完成后，统一将 `dev` 分支合并到 `test` 分支测试，测试通过后，合并成新 `feature` 分支； 
3. 对于发生在 `feature` 中的紧急问题，从当前 `feature` 分支新建 `hotfix` 分支进行修复，测试后合并回 `feature` 发布。 
## 提交信息规范 
> 提交信息前缀，可快速定位。 
- `feat`: 新功能 
- `fix`: 修复问题 
- `perf`: 代码优化 
- `docs`: 文档更新 
- `style`: 代码格式调整（不影响代码） 
- `refactor`: 重构 
- `test`: 测试相关 
- `chore`: 构建过程或辅助工具变动

## 文件和文件夹命名规范
> 遵循大厂标准，确保跨平台兼容性和可维护性。

### 文件夹命名规范
- **统一使用英文命名**：所有文件夹名称必须使用英文，禁止使用中文字符，确保跨平台兼容性
- **使用 kebab-case（小写+连字符）**：多单词使用连字符分隔，如 `data-structures`、`web-development`
- **简洁明确**：文件夹名称应清晰反映其内容，便于快速定位
- **避免特殊字符**：禁止使用系统保留字符 `\/:*?"<>|`，避免空格
- **示例**：
  - ✅ `cpp`、`data-structures`、`linux`、`web-development`
  - ❌ `数据结构`、`C++学习`、`Web Development`（含空格）

### 文件命名规范
- **统一格式**：文件名应遵循统一格式，包含关键信息
- **使用下划线或连字符分隔**：建议使用下划线 `_` 或连字符 `-` 分隔不同字段，避免使用空格
- **时间前置**：重要文档建议将日期放在前面，便于排序和追溯，格式：`YYYY-MM-DD`
- **模块化结构**：文件名由多个字段组成，格式：`日期_项目名_内容描述_版本_作者`
- **禁止特殊字符**：避免使用系统保留字符 `\/:*?"<>|`，确保跨平台兼容性
- **大小写一致性**：建议使用小写字母，保持命名风格一致
- **示例**：
  - ✅ `2025-12-31_project-report_v1.0_shihonggan.md`
  - ✅ `cpp_basic-syntax_notes.md`
  - ✅ `data-structures_binary-tree_summary.md`
  - ❌ `项目报告 2025.md`（含中文和空格）
  - ❌ `C++学习笔记.docx`（含特殊字符）
